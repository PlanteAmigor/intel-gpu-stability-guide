## Analysis & Root Cause

### 1. Why NaN appears with long durations

NaN in diffusion models for long audio is a **latent-space drift** problem, not an out-of-memory issue (you have 128 GB RAM / ~115 GB shared VRAM):

| Factor | Explanation |
|--------|-------------|
| **Training distribution** | The DiT and VAE are trained on audio clips of limited duration. Beyond that range, the latent representations can drift outside the trained distribution. |
| **Attention instability** | Longer sequences produce larger attention logits, increasing the risk of softmax overflow and numerical divergence. |
| **Positional encoding saturation** | Positional embeddings may not generalize well to sequence lengths far beyond those seen during training. |
| **VAE decoder sensitivity** | Once latents drift outside the expected distribution, the VAE decoder can produce extreme values → NaN in the final waveform. |

### 2. Why Windows crashes harder than Linux (240s vs 600s)

| Platform | Speed | Async submit rate | Software stack maturity | NaN threshold |
|----------|-------|-------------------|------------------------|---------------|
| Windows 11 | 11–12 tok/s | Dense | Intel Windows XPU driver also immature | Crash/reboot at ~240s |
| Ubuntu 26.04 | 4–5 tok/s | Sparse | Intel Linux XPU = very new (2025) | NaN at ~600s, no crash |

- The root cause is **Intel XPU software stack immaturity** — both Windows and Linux drivers are new and lack CUDA's 15+ years of protection mechanisms (TDR, OOM retry chain, expandable segments, async error reporting).
- Windows crashes harder (system reboot) not because of thermal issues, but because the higher token rate means **more frequent async submissions** → faster accumulation of Level Zero driver state inconsistencies → more catastrophic failure mode.
- On Linux, slower token rate means fewer operations per second, so it takes longer to trigger the same software bugs.
- **Key evidence**: On Ubuntu, 600s NaN followed by a clean 300s generation proves GPU hardware is perfectly fine. The issue is purely software — the XPU driver stack enters an inconsistent state under sustained load.

### 3. Why "NaN cascade" on Windows but not on Linux

On Windows, NaN triggers a **driver-level crash** (assertion failure / reboot) — the Level Zero driver state is corrupted beyond recovery.

On Linux, NaN is a **soft failure** — the tensor contains garbage but the Level Zero driver remains functional. A `synchronize()` + `empty_cache()` resets the driver state, and the next generation starts clean.

### 4. Suggested mitigations

| Mitigation | How to apply |
|------------|-------------|
| **Add active cooling** | DiT every 2 steps 3s + LLM every 50 steps 5s + VAE decode every 50 chunks 5s (verified working) |
| **Use shorter durations without cooling** | ≤ 300s (5 min) without active cooling |
| **Monitor memory fragmentation** | Watch `inactive_split_bytes` in `torch.xpu.memory_stats()` |
| **Wait for Intel XPU software maturity** | Intel is actively improving — future releases may close the protection gap |
| **Track protection gap in intel_xpu_with_cuda.md** | Covers all missing CUDA protections in detail |

### 5. Key takeaway (revised based on 600s success)

> **The NaN at long durations is NOT a hardware defect — it's caused by Intel's immature XPU software stack lacking CUDA's protection mechanisms.**
>
> With active cooling to compensate for missing TDR, OOM retry, and memory management, the iGPU can reliably generate 600s+ audio on Linux.
>
> **Update 2026-05-30**: Successfully generated 600s (10 min) audio after adding tiered cooling. Previous 480s limit was conservative — the actual limit is software-bound, not hardware-bound.

## 分析 & 根本原因

### 1. 为什么长音频会出现 NaN

长音频 NaN 是**潜变量空间漂移**问题，不是显存不够（你有 128GB 内存/约 115GB 共享显存）：

| 因素 | 说明 |
|------|------|
| **训练分布限制** | DiT 和 VAE 训练时使用的音频时长有限，超出范围的潜变量会偏离训练分布 |
| **注意力不稳定** | 长序列产生更大的 attention logits，增加 softmax 溢出的风险 |
| **位置编码饱和** | 位置嵌入对远超训练时长的序列泛化能力有限 |
| **VAE 解码器敏感** | 潜变量漂移后，VAE 解码器可能输出极端值 → 最终波形出现 NaN |

### 2. 为什么 Windows 比 Linux 更容易崩溃（240s vs 600s）

| 平台 | 速度 | 驱动激进程度 | 热/功耗压力 | NaN 出现阈值 |
|------|------|------------|------------|-------------|
| Windows 11 | 11–12 tok/s | 高（更高频率） | 高 → ~240s 崩溃/重启 | 更低 |
| Ubuntu 26.04 | 4–5 tok/s | 低（保守调度） | 低 → 稳定到 ~600s | 更高 |

- Windows 驱动让 iGPU 跑在更高性能，持续负载下功耗/温度升高 → 触发驱动崩溃、断言失败甚至系统重启
- Linux 上同样负载跑得更慢但更凉快，所以系统能撑更久。NaN 在 600s 仍然出现，但只是**纯数值问题**，不会导致系统级崩溃
- **重要证据**: Ubuntu 上 600s NaN 后下一次 300s 生成完全正常 — 说明 GPU 硬件没问题，NaN 是软件/数值问题

### 3. 为什么 Windows 上出现 "NaN 级联"，Linux 不会

- **Windows**: NaN 导致**系统级崩溃**（断言失败/重启）→ GPU 驱动状态损坏 → 后续生成必定失败
- **Linux**: NaN 是**软故障** — 张量里有 NaN，但 GPU 驱动状态健康 → 下一次生成重置计算图，正常运行

### 4. 缓解方案（不改代码，仅配置）

| 方案 | 操作 |
|------|------|
| **控制时长** | Windows 上 ≤ 360s（6 分钟），Linux 上 ≤ 480s（8 分钟） |
| **开启 `velocity_norm_threshold`** | 如果 UI 中有此选项，设为 2.0–3.0，可在扩散过程中钳制异常预测值 |
| **开启 INT8 量化** | 减少内存带宽压力，有助于数值稳定 |
| **降低 Windows 热压力** | 用 Intel Arc Control 降低功耗限制或 GPU 频率 |
| **等待 Intel Linux XPU 驱动更新** | Intel 正在持续优化 Linux XPU 性能，未来驱动可能缩小速度差距 |

### 5. 核心结论（已修正）

> **长音频 NaN 不是模型限制（600s 已成功验证），也不是硬件缺陷。根本原因是 Intel XPU 软件栈太新——Level Zero 驱动、PyTorch XPU 分配器、内存管理都缺少 CUDA 十几年来积累的保护机制。**
>
> 冷却之所以有效，不是因为 GPU 过热，而是因为它**手动补偿了 Intel 软件栈缺失的 TDR、OOM 重试链、异步错误处理等保护**。每次 `synchronize()` + `empty_cache()` 相当于给 XPU 驱动一个「健康检查点」。
>

**原先猜想 2: 位置编码饱和**
- 状态：❌ **被推翻**
- 依据：同上。

**原先猜想 3: GPU 过热/功耗问题**
- 状态：❌ **推翻（表述偏差）**
- 纠正：问题根源不是硬件过热，而是 **Intel XPU 软件栈太新、不够完善**。冷却之所以有效，是因为它给了 XPU 驱动/运行时恢复状态的机会——CUDA 有 TDR 看门狗、OOM 重试链、expandable segments 等十几年的软件积累，XPU 这些全都没有。冷却本质上是**手动补偿 Intel 软件栈缺失的保护机制**，不是给硬件降温。
- 依据：ARC iGPU 在 Windows 上跑 11-12 token/s 不崩（说明硬件能承受更高负载），Windows 崩溃更可能是 **Intel Windows XPU 驱动也有类似的软件问题**。

### 新猜想

**新猜想 1: Intel XPU 内存分配器缺少碎片整理导致渐进式崩溃**
- 从 `intel_xpu_with_cuda.md` 的对比可知：CUDA 有 expandable segments + 6 阶段 OOM 重试链 + 自动碎片整理，XPU 全都没有。
- LLM 生成中 KV Cache 从 0 渐进增长到 ~1900 tokens 时 OOM，是内存碎片累积到临界点后分配器无法找到连续块，而非物理显存耗尽（你有 115GB 共享显存）。
- 冷却期间 `empty_cache()` + `gc.collect()` 释放碎片段，重置了分配器状态。

**新猜想 2: XPU Level Zero 驱动的同步/异步状态管理问题**
- CUDA 有 `DeferredCudaCallError` 延迟报告异步错误，XPU 的 Level Zero 驱动可能在长时间异步提交队列中积累状态不一致。
- 冷却中的 `torch.xpu.synchronize()`（隐含在 `empty_cache` 中）给了驱动机会清空队列、恢复干净状态。
- Windows 下更快的速度（11 vs 4-5 tok/s）意味着异步提交更密集，状态不一致更快累积，所以崩溃更早（240s vs 600s）。

**新猜想 3: Level Zero 驱动缺乏长时间任务的健康检查机制**
- CUDA 驱动有 TDR（Timeout Detection & Recovery），内核执行超时后自动重置上下文并报错，不会死锁。
- Intel Level Zero 驱动缺少等效机制。长时间无同步的推理队列一旦进入异常状态，只能靠外部干预恢复。
- 冷却中的 `synchronize()` / `empty_cache()` 充当了手动健康检查点。

### 当前配置总结

成功的关键变化是添加了分层冷却机制：

| 阶段 | 冷却策略 |
|------|---------|
| **LLM 生成** | 每 50 token 休息 5s + 降频检测（超基线 2x 额外 10s） |
| **DiT 扩散** | 每 2 步休息 3s |
| **VAE 解码** | 每 50 块休息 5s + 降频检测（本次新增） |

### 更新后的结论

> **600s 的成功表明：Intel Arc iGPU + Linux XPU 足够稳定以完成 10 分钟音频生成，前提是加入主动冷却来补偿 Intel 软件栈缺失的保护机制。**
>
> 之前的 NaN 不是模型限制，也不是硬件缺陷，而是 **Intel XPU 软件栈（驱动、Level Zero 运行时、PyTorch XPU 分配器）太新，缺少 CUDA 十几年积累的保护机制**（TDR、OOM 重试链、expandable segments、异步错误报告等）。冷却本质上是在手动实现这些缺失的保护。
